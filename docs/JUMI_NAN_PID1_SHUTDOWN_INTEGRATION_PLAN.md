# JUMI nan PID 1 Shutdown Integration Plan

문서 상태: Draft v0.1
작성일: 2026-07-01
대상 프로젝트: `JUMI` / `node-artifact-runtime(nan)`

## 결론

JUMI는 DAG node Pod에서 `nan`을 runtime commandlet으로 실행할 때,
`nan`이 직접 PID 1로 동작할 수 있다는 전제를 지원해야 한다.

JUMI 쪽 기본 실행 모델은 다음이다.

```text
Pod main container
  PID 1: /bin/sh wrapper generated by JUMI compatibility path
    exec /usr/local/bin/nan run --contract /jumi/node-contract.json -- <user command>
```

최종 목표는 shell wrapper를 제거하고 아래 모델로 가는 것이다.

```text
Pod main container
  PID 1: /usr/local/bin/nan
    nan run --contract /jumi/node-contract.json -- <user command>
```

`tini`는 optional hardening layer로만 취급한다.

```text
/usr/bin/tini -- /usr/local/bin/nan run --contract /jumi/node-contract.json -- <user command>
```

필수 runtime contract는 `nan` 직접 실행이다.

## JUMI 책임

JUMI는 아래를 보장해야 한다.

```text
- node contract JSON 생성
- user command를 nan run 뒤에 argv 그대로 전달
- JUMI_SHUTDOWN_GRACE_PERIOD 주입
- JUMI_RUN_TIMEOUT 주입 또는 Kubernetes activeDeadlineSeconds와 정책 정합성 유지
- Pod terminationGracePeriodSeconds를 nan grace period보다 크게 설정
- timeout/interrupted/killed termination summary를 node failure로 해석
- timeout/interrupted/killed 경로에서 manifest 미존재를 정상 실패 경로로 취급
- succeeded 전환은 manifest 회수와 required artifact validation 이후에만 수행
```

JUMI가 담당하지 않는 것:

```text
- nan 내부 process group kill
- nan 내부 child subreaper/reap
- tool-specific cleanup hook
- AH RegisterArtifact를 nan에 위임
```

## Configuration Contract

권장 기본값:

```text
JUMI_SHUTDOWN_GRACE_PERIOD=25s
terminationGracePeriodSeconds=30
```

관계는 반드시 아래를 만족해야 한다.

```text
JUMI_SHUTDOWN_GRACE_PERIOD < terminationGracePeriodSeconds
```

JUMI가 PodSpec을 만들 때는 `terminationGracePeriodSeconds`를 고정값으로만 두지 말고,
주입된 `JUMI_SHUTDOWN_GRACE_PERIOD`보다 크게 계산해야 한다.

```text
terminationGracePeriodSeconds = ceil(JUMI_SHUTDOWN_GRACE_PERIOD seconds) + 5
```

JUMI가 `JUMI_RUN_TIMEOUT`을 넣는 경우 timeout은 `nan`이 해석한다.

```text
JUMI_RUN_TIMEOUT=30m
```

Kubernetes `activeDeadlineSeconds`를 같이 사용하는 경우에는 아래 관계가 안전하다.

```text
JUMI_RUN_TIMEOUT + JUMI_SHUTDOWN_GRACE_PERIOD < activeDeadlineSeconds
```

그래야 `nan`이 먼저 timeout summary를 쓰고 process group을 정리할 시간이 생긴다.

## PodSpec Requirements

JUMI K8s Job 생성 경로는 PodSpec에 아래를 반영해야 한다.

```yaml
spec:
  template:
    spec:
      restartPolicy: Never
      terminationGracePeriodSeconds: 30 # 25s nan grace + 5s margin
      containers:
        - name: main
          env:
            - name: JUMI_SHUTDOWN_GRACE_PERIOD
              value: 25s
```

현재 Job 생성 위치:

```text
pkg/spawner/k8s_jobclient.go
```

현재 command wrapping 위치:

```text
pkg/backend/spawner_k8s.go
```

현재 runtime env 주입 위치:

```text
pkg/executor/executor.go
```

## Command Injection Direction

현재 compatibility path는 shell wrapper가 contract file을 쓴 뒤 `exec nan run` 한다.

```text
/bin/sh -ceu <write contract; exec nan run ...> jumi-node-contract <user command>
```

이 경로는 `exec`를 사용하므로 shell wrapper가 최종적으로 `nan`으로 교체된다.
따라서 실제 command 실행 시점에는 `nan`이 PID 1에 가까운 역할을 한다.

최종 목표:

```text
Command:
  - /usr/local/bin/nan
Args:
  - run
  - --contract
  - /jumi/node-contract.json
  - --
  - <user command argv...>
```

단, contract file을 누가 쓰는지 정리되기 전까지는 현재 compatibility wrapper를 유지할 수 있다.

## Failure Semantics

JUMI는 `nan` termination summary를 다음처럼 해석한다.

```text
status=timeout       -> node failed, reason timeout
status=interrupted   -> node failed/cancelled, reason external termination
status=killed        -> node failed, reason forced kill after grace period
status=command_failed -> node failed, preserve user exit code
status=inspect_failed -> node failed, artifact contract violation
```

아래 상태에서는 manifest가 없어도 JUMI가 별도 manifest 회수 오류로 덮어쓰면 안 된다.

```text
timeout
interrupted
killed
command_failed
materialization_failed
invalid_config
```

반대로 `nan`이 성공으로 종료했는데 required manifest가 없으면 fail-closed 해야 한다.

## Smoke Test Plan

### Smoke 1: timeout kills shell child and grandchild

목표:

```text
nan timeout이 shell grandchild까지 종료하고 manifest를 쓰지 않는다.
```

fixture command:

```sh
sh -c 'sleep 600 & wait'
```

JUMI runtime env:

```text
JUMI_RUN_TIMEOUT=3s
JUMI_SHUTDOWN_GRACE_PERIOD=2s
```

PodSpec:

```text
terminationGracePeriodSeconds=10
```

Acceptance:

```text
- Job fails
- termination log status is timeout
- artifact manifest does not exist or is not treated as success
- no long-running sleep process remains in the container after termination
```

### Smoke 2: child ignores SIGTERM and gets SIGKILL

fixture command:

```sh
sh -c "trap '' TERM; sleep 600"
```

Acceptance:

```text
- Job finishes before Kubernetes terminationGracePeriodSeconds
- termination log status is timeout
- nan exits with ExitTimeout
- no success manifest is registered
```

### Smoke 3: external Pod deletion

fixture command:

```sh
sh -c 'sleep 600'
```

Procedure:

```text
1. Submit node Job
2. Wait until Pod Running
3. Delete Pod or Job
4. Observe terminal state
```

Acceptance:

```text
- nan receives SIGTERM
- nan forwards signal to user process group
- termination summary is interrupted or killed
- JUMI does not register artifacts
```

### Smoke 4: success path remains unchanged

fixture command:

```sh
sh -c 'echo ok > /out/report'
```

Acceptance:

```text
- Job succeeds
- nan writes manifest
- JUMI reads manifest
- JUMI registers required artifact
```

## Implementation Tasks

### Phase 1: JUMI runtime env defaults

```text
- inject JUMI_SHUTDOWN_GRACE_PERIOD when missing
- define JUMI default value in one place
- ensure user-provided env can override only if policy allows it
```

### Phase 2: PodSpec grace period

```text
- set terminationGracePeriodSeconds on node Job Pods
- compute value greater than JUMI_SHUTDOWN_GRACE_PERIOD, including user overrides
- add unit test around buildK8sJob
```

### Phase 3: timeout policy alignment

```text
- decide whether JUMI_RUN_TIMEOUT maps from node.TimeoutPolicy
- avoid racing nan timeout against Kubernetes activeDeadlineSeconds
- document precedence between JUMI timeout and Kubernetes deadline
```

### Phase 4: smoke fixtures

```text
- add timeout grandchild smoke
- add SIGTERM-ignoring child smoke
- add external deletion smoke
- keep success manifest smoke as regression baseline
```

## Immediate Code Touch Points

```text
pkg/executor/executor.go
  injectRuntimeContextEnv should add JUMI_SHUTDOWN_GRACE_PERIOD default.

pkg/backend/spawner_k8s.go
  toAttemptRequest should map node timeout policy to JUMI_RUN_TIMEOUT if nan owns timeout.

pkg/spawner/k8s_jobclient.go
  buildK8sJob should set terminationGracePeriodSeconds.

scripts/run-jumi-*-smoke*.sh
  add smoke entrypoints or extend existing runtime smoke scripts.
```

## Open Decisions

```text
- Should users be allowed to override JUMI_SHUTDOWN_GRACE_PERIOD per node?
- Should JUMI_RUN_TIMEOUT be the only timeout authority for node runtime commands?
- Should Kubernetes activeDeadlineSeconds remain disabled unless explicitly requested?
- Should final command injection remove the shell wrapper now or after contract file mounting is redesigned?
```

## Current Recommendation

Proceed in this order:

```text
1. JUMI injects JUMI_SHUTDOWN_GRACE_PERIOD=25s by default.
2. K8s Job PodSpec sets terminationGracePeriodSeconds=30.
3. JUMI maps node timeout policy to JUMI_RUN_TIMEOUT only when nan runtime-helper mode is active.
4. Add smoke fixtures for timeout and external termination.
5. Remove shell compatibility wrapper only after contract file delivery is no longer shell-generated.
```
